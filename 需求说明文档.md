## 1. 项目概述

### 1.1 项目目标

构建一套自动化销售线索发现系统，面向免疫荧光领域，每日自动从论文库和招标公告中抓取、提取、评级潜在客户信息，替代销售团队的手工检索工作。

### 1.2 核心价值

- 每日自动发现新线索，覆盖论文发表和设备采购两大信号源
- 多维度自动评分，帮助销售聚焦高优线索（A 级）
- 通过反馈闭环持续优化线索质量

### 1.3 使用流程

```
系统每日 06:00 自动运行
    → 搜索发现新论文/招标公告
    → 抓取页面内容
    → LLM 提取结构化字段
    → 去重 + 补齐 + 评分
    → 导出 CSV（每日增量 / 每周日全量）
    → 销售手动导入飞书多维表格
    → 销售使用并标注反馈
    → 反馈回流优化策略（人工审核后执行）
```

## 4. 数据抓取需求

### 4.1 技术路线（已确认）

| 层级 | 工具 | 用途 |
|------|------|------|
| 主力 | Jina Reader API (`r.jina.ai`) | 将页面 URL 转为干净 Markdown |
| 搜索 | Jina Search API (`s.jina.ai`) | 用关键词搜索发现新 URL |
| 兜底 | Playwright | Jina 无法访问的页面（验证码/反爬） |
| 提取 | Claude API | 从 Markdown 中提取结构化字段为 JSON |

### 4.2 增量控制

- 以 URL 为唯一标识，已爬过的 URL 记录在 `crawled_urls` 表中
- 每日任务只处理新发现的、未爬过的 URL
- 避免资源浪费和重复入库

### 4.3 去重规则

| 层级 | 规则 |
|------|------|
| 数据库级 | 同一 URL 不重复入库 |
| 数据整合级 | 相同 `[姓名 + 单位 / 联系方式 / 地址]` 只输出最新一条 |

### 4.4 缺失字段补齐

通常缺失的是联系方式（邮箱/手机）。

补齐来源优先级（需与 shane 确认可行性后实施）：
1. 论文通讯作者邮箱、电话（从论文页面提取）
2. 单位官网查询
3. 公开网页搜索（`s.jina.ai`）

### 4.5 异常处理

| 异常类型 | 处理方式 |
|----------|----------|
| 网站不可用 / 超时 | 重试 1 次 |
| 验证码 | 立即通知 shane，不自动处理 |
| 失败数据 | 当日不入库，次日重试 |
| 日志 | 保留 30 天 |

## 5. 线索评级需求

### 5.1 评级体系

- 内部保留数值分（0-100），飞书展示 ABCD 等级
- **论文和招标使用两套独立评分规则**

| 等级 | 分数区间 | 含义 |
|------|----------|------|
| A | ≥ 80 | 高优线索 |
| B | 65 – 79 | 中优线索 |
| C | 50 – 64 | 一般线索 |
| D | < 50 | 低优线索 |

### 5.2 论文线索评分维度（满分 100）

| 维度 | 权重 | 满分 | 高分条件 | 中分条件 | 低分条件 |
|------|------|------|----------|----------|----------|
| A 信息完备度 | 20% | 20 | 手机+邮箱+地址齐全：20 | 缺手机号：12 | org_only：5 |
| D 匹配度 | 25% | 25 | 核心关键词+关联设备：25 | 泛关键词：15 | 弱匹配：5 |
| E 时效性 | 20% | 20 | 最近90天：20 | 90-180天：12 | 更早：5 |
| C 决策链条 | 15% | 15 | PI+通讯作者完整：15 | 仅联系人：9 | 仅单位：3 |
| G 机构类型 | 10% | 10 | 985/211/双一流/科研院所：10 | 三甲医院：7 | 企业/其他：4 |
| F 历史互动 | 10% | 10 | 已成交/已报价：10 | 无互动：5 | 已流失：0 |

> 论文数据不适用「B 预算状态」维度，其权重已分配至匹配度（+5%）和时效性（+5%）。

### 5.3 招标线索评分维度（满分 100）

| 维度 | 权重 | 满分 | 高分条件 | 中分条件 | 低分条件 |
|------|------|------|----------|----------|----------|
| B 预算状态 | 20% | 20 | 已明确预算/立项：20 | 有预算线索：12 | 经费不详：5 |
| D 匹配度 | 20% | 20 | 核心关键词+关联设备：20 | 泛关键词：12 | 弱匹配：4 |
| E 时效性 | 20% | 20 | 最近90天：20 | 90-180天：12 | 更早：5 |
| A 信息完备度 | 15% | 15 | 手机+邮箱+地址齐全：15 | 缺手机号：9 | org_only：3 |
| C 决策链条 | 10% | 10 | PI/联系人/设备科完整：10 | 仅联系人：6 | 仅单位：2 |
| G 机构类型 | 10% | 10 | 985/211/双一流/科研院所：10 | 三甲医院：7 | 企业/其他：4 |
| F 历史互动 | 5% | 5 | 已成交/已报价：5 | 无互动：2 | 已流失：0 |

### 5.4 机构分类参考

| 类别 | 说明 | 评分档位 |
|------|------|----------|
| 985/211/双一流 | 顶级高校 | 最高 |
| 科研院所 | 中科院系统等 | 最高 |

## 6. 数据输出需求

### 6.1 输出格式

两张独立的 CSV 表：**论文线索表** 和 **招标线索表**

### 6.2 输出周期

| 周期 | 类型 | 规则 |
|------|------|------|
| 每日 | 增量 | 仅新增和变更的线索 |
| 每周日 | 全量 | 先与数据库同步，diff 字段标注变更标记 |

### 6.3 全量导出的 diff 标注

每周日全量导出时，每条记录增加 `变更标记` 字段：
- `新增`：本周新入库的线索
- `已更新`：本周有字段变更的线索
- `无变化`：与上次全量导出一致

目的：防止销售在全量表中无法分辨哪些是新变化。

## 7. 销售反馈与优化循环

### 7.1 反馈输入方式

销售将反馈数据导出为 CSV，放入项目 `feedback/` 目录。

- 文件命名：`feedback_YYYY-MM-DD.csv`
- 通过命令触发系统读取并处理

### 7.2 反馈状态

```
未处理 → 已联系 → 已拜访 → 已报价 → 已成交
                                  ↘ 暂缓跟进
                                  ↘ 已流失
```

### 7.3 反馈维度

| 维度 | 说明 |
|------|------|
| 线索准确性 | 信息是否正确 |
| 需求匹配度 | 客户是否真有需求 |
| 联系方式有效性 | 能否联系到人 |
| 成交速度 | 从接触到成交的周期 |
| 成交价格 | 成交金额 |

### 7.4 反馈回流处理

采用 **模式 (a)：人工审核模式**

1. 系统分析反馈数据，生成优化建议报告
2. 报告内容：哪些关键词带来高质量线索、哪些数据源转化率高、评分权重调整建议
3. shane 审核后确认执行
4. 执行后生成新策略版本

### 7.5 策略版本管理

| 规则 | 说明 |
|------|------|
| 版本号 | v1, v2, v3... 递增 |
| 切换新版本 | 旧版本归档，不再产出新数据 |
| 回退 | 支持一键回退到上一版本 |
| 版本记录 | 变更原因、变更内容、变更时间、操作人 |
| 配置快照 | 每个版本保存完整配置快照（JSONB） |

## 9. 运行与监控

### 9.1 每日任务流程

```
06:00  启动
  ├── 用 s.jina.ai 搜索各数据源新 URL
  ├── 过滤已爬 URL（查 crawled_urls 表）
  ├── 用 r.jina.ai 读取新页面内容
  ├── 用 Claude API 提取结构化字段
  ├── 入库（去重 + 标准化）
  ├── 运行评分引擎
  ├── 导出增量 CSV
  └── （周日额外导出全量 CSV + diff 标注）
10:00  完成
```

### 9.2 通知

| 事件 | 通知方式 |
|------|----------|
| 任务完成 | 飞书机器人（后续配置） |
| 异常/验证码 | 飞书机器人（后续配置） |
| MVP 阶段 | 日志文件 |

### 9.3 日志

- 使用 Python `logging` 标准库
- 保留 30 天，自动清理
- 级别：DEBUG / INFO / WARNING / ERROR

## 11. 约束与禁止项

### 11.1 配置修改约束

- 所有配置变更必须通过 Agent Chat 进行
- 每次变更记录原因和版本号

### 11.2 需先通知 shane

- 修改评分权重或等级阈值
- 新增或移除数据源
- 变更数据库 schema
- 确定缺失字段补齐方案
- API 费用超预期
- 遭遇验证码

### 11.3 绝对禁止

- ❌ 删除数据库或数据表
- ❌ `DROP TABLE`、`TRUNCATE`、批量 `DELETE`
- ❌ 未告知修改评分权重
- ❌ 提交含 API Key / 密码的代码
- ❌ 生产库上直接测试
- ❌ 修改日志保留策略
