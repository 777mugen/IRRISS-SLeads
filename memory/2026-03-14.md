## 📊 数据管道架构

### 分层设计

```
┌─────────────────┐
│  Jina Reader    │ → 保留所有原始数据（图片、链接、完整HTML结构）
│  (Raw Data)     │
└─────────────────┘
        ↓
┌─────────────────┐
│  raw_markdown   │ → 存储原始 Markdown（包含所有信息）
│  (Storage)      │
└─────────────────┘
        ↓
┌─────────────────┐
│  智谱批处理      │ → 提取结构化信息（标题、作者、地址等）
│  (Extraction)   │
└─────────────────┘
        ↓
┌─────────────────┐
│  paper_leads    │ → 存储结构化数据（提取后的信息）
│  (Storage)      │
└─────────────────┘
        ↓
┌─────────────────┐
│  评分/导出/应用  │ → 业务逻辑（评分、导出CSV、Feishu通知）
│  (Business)     │
└─────────────────┘
```

### 职责划分

**Jina Reader (Raw Data)**:
- 职责: 爬取并保留所有原始数据
- 原则: 不删除任何信息
- 可删除: 导航栏、广告等无关元素（不影响数据提取）

**智谱批处理 (Extraction)**:
- 职责: 从原始数据中提取结构化信息
- 原则: 从文本中提取信息，忽略图片链接
- 输出: JSON 格式的结构化数据

**解析器 (Processing)**:
- 职责: 清洗、验证、转换格式
- 原则: 数据清洗在应用层更灵活
- 输出: 符合数据库约束的数据

**业务层 (Business)**:
- 职责: 评分、导出、应用逻辑
- 原则: 业务规则在此层实现
- 输出: CSV、通知等

## 🎓 经验教训

### 1. 原始数据的重要性

**问题**: 为了减少 token 消耗，过早删除了图片和链接

**后果**: 丢失了关键信息（作者姓名、机构地址）

**教训**: 
- ✅ 原始数据一旦丢失，无法恢复
- ✅ 不同的处理需求可能需要不同的数据
- ✅ 后续处理可以灵活调整，但原始数据必须完整

### 2. 用户反馈的价值

**用户**: 董胜豪手动测试并提供了完整的内容

**发现**: Jina 本身可以提取完整信息，问题在于 API 参数

**教训**:
- ✅ 用户的反馈和测试非常宝贵
- ✅ 系统性问题需要用户参与验证
- ✅ 提供清晰的问题复现步骤有助于快速定位

### 3. 架构原则的重要性

**背景**: 项目初期没有明确的架构原则

**结果**: 各个环节职责不清，数据处理混乱

**教训**:
- ✅ 及早确立架构原则
- ✅ 文档化并持续更新
- ✅ 团队成员都要遵守
