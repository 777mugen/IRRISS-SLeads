### 参数配置优化

**新增 Headers**:
```python
headers = {
    'Authorization': 'Bearer {api_key}',
    'Accept': 'text/plain',
    'X-Respond-With': 'markdown',
    'X-Respond-Timing': 'resource-idle',      # 平衡速度和完整性
    'X-Timeout': '60',
    'X-Engine': 'browser',                    # 模拟浏览器，减少反爬
    'X-Cache-Tolerance': '3600',              # 1小时缓存

    'X-Retain-Images': 'none',                # ✅ 去除图片
    'X-With-Generated-Alt': 'false',
    'X-Locale': 'en-US',
    'X-Referer': 'https://doi.org/',          # 模拟从 DOI 跳转
    'X-Token-Budget': '50000',
    'X-Robots-Txt': 'false'
}
```

### 关键改进

- ✅ `X-Retain-Images: none` - 去除图片（用户要求）
- ✅ `X-Respond-Timing: resource-idle` - 平衡速度（比 network-idle 快）
- ✅ `X-Cache-Tolerance: 3600` - 利用缓存提升速度

## 🧠 数据同步策略 Brainstorm（21:54）

### 问题
- **raw_markdown**: 20 条记录（19 completed, 1 processing）
- **paper_leads**: 0 条记录 ⚠️

### 两个方案对比

#### **方案 1**: 先创建 raw_markdown，智谱提取后更新 paper_leads
**流程**:
```
查 DOI → 创建 raw_markdown (status=pending)
爬 raw 数据 → 更新 markdown_content
送智谱提取 → 更新 (status=processing, batch_id)
提取完成 → 检查 paper_leads 是否存在
  - 不存在 → 创建 paper_leads
  - 已存在 → 更新 paper_leads
更新 raw_markdown (status=completed)
```

**优点**:
- ✅ 简单直接
- ✅ 状态清晰（raw_markdown 负责跟踪处理流程）
- ✅ 容错性好（智谱提取失败不影响 paper_leads）
- ✅ 支持重试（根据 status 重新处理失败任务）
- ✅ 符合数据库约束（paper_leads 有必填字段，不能提前创建）

**缺点**:
- ⚠️ 延迟创建（paper_leads 记录创建晚于 raw_markdown）
- ⚠️ 数据不一致期（智谱提取完成前，两个表不完全对应）

#### **方案 2**: 在创建 raw_markdown 时同步创建 paper_leads
**流程**:
```
查 DOI → 创建 raw_markdown (status=pending)
       → 创建 paper_leads (只有 doi, source_url, status=processing)
爬 raw 数据 → 更新 markdown_content
送智谱提取 → 更新 (status=processing, batch_id)
提取完成 → 直接更新 paper_leads
更新 raw_markdown (status=completed)
```

**优点**:
- ✅ 数据一致性（两个表始终完全对应）
- ✅ 查询简单（可以直接从 paper_leads 查询所有论文）

**缺点**:
- ⚠️ 提前创建（paper_leads 在没有数据时就创建了）
- ⚠️ 脏数据风险（智谱提取失败，paper_leads 会有空记录）
- ⚠️ **字段约束冲突**（paper_leads 有必填字段，提前创建会违反数据库约束）❌
- ⚠️ 逻辑耦合（两个表的创建逻辑耦合在一起）

### 关键问题
**paper_leads 表有必填字段**:
- title（标题）✅
- published_at（发表时间）✅
- article_url（原文链接）✅
- source（来源）✅
- name（通讯作者）✅
- address（单位地址）✅
- phone（联系电话）✅
- email（电子邮箱）✅

**方案 2 的问题**: 在创建 paper_leads 时，这些字段都还没有，会违反数据库约束！

### 推荐方案
**方案 1 + 优化** ⭐

**优化点**:
1. 统一创建逻辑：在智谱提取完成时，检查 paper_leads 是否存在，不存在则创建
2. 状态跟踪：在 raw_markdown 表中跟踪处理状态
3. 错误处理：如果智谱提取失败，raw_markdown 标记为 failed，不创建 paper_leads

**优化后的流程**:
```
1. 查 DOI → 创建 raw_markdown (status=pending)
2. 爬 raw 数据 → 更新 raw_markdown.markdown_content
3. 送智谱提取 → 更新 raw_markdown (status=processing, batch_id)
4. 提取完成 → 
   - 检查 paper_leads 是否存在 (通过 doi)
   - 不存在 → 创建 paper_leads (包含所有必填字段)
   - 已存在 → 更新 paper_leads
   - 更新 raw_markdown (status=completed)
5. 提取失败 → 
   - 更新 raw_markdown (status=failed, error_message)
   - 不创建 paper_leads
```

### 决策矩阵

| 维度 | 方案 1 | 方案 2 | 方案 1 + 优化 |
|------|--------|--------|---------------|
| **简单性** | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| **数据一致性** | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| **容错性** | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| **查询复杂度** | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| **字段约束** | ⭐⭐⭐ | ❌ | ⭐⭐⭐ |
| **状态跟踪** | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |

### 理由
1. ✅ 符合数据库设计：paper_leads 有必填字段，不能提前创建
2. ✅ 容错性好：智谱提取失败不影响 paper_leads
3. ✅ 状态清晰：raw_markdown 负责跟踪处理流程
4. ✅ 支持重试：可以重新处理失败的任务
5. ✅ 数据一致性：通过统一逻辑保证两个表对应
