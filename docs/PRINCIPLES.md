# IRRISS-SLeads 项目原则

**最后更新**: 2026-03-14
**决策人**: 董胜豪

## 核心原则

### 1. 准确性优先 ⭐

**原则**: 准确性 > Token 消耗

**原因**: 
- 太长的无效信息会带来模型的**注意力问题**，降低提取准确性
- 不是为了节省 token，而是为了提高准确性
- 无效信息（如论文的 Methodology, Results, Discussion）会干扰模型识别

**应用**:
- ✅ 使用"精准截断"策略，只提取元数据部分
- ✅ 去除无关的学术内容
- ❌ 不要只为了节省 token 而截断

---

### 2. 迭代优化循环 🔄

**原则**: 爬取 → 检查 → 优化 → 总结 → 优化的循环

**原因**:
- 期刊格式不确定，需要根据实际情况调整
- 没有通用的最优方案，需要在实践中迭代
- 每一次爬取都是学习机会

**应用**:
- ✅ 爬取一批论文 → 检查提取结果 → 发现问题 → 优化策略
- ✅ 总结期刊格式特点 → 优化截断规则 → 优化 Prompt
- ✅ 持续循环，逐步提高准确性
- ❌ 不要试图一次性找到完美方案

**迭代流程**:
```
┌─────────────────┐
│  爬取论文        │
└─────────────────┘
        ↓
┌─────────────────┐
│  检查提取结果    │
└─────────────────┘
        ↓
┌─────────────────┐
│  定位问题        │
└─────────────────┘
        ↓
┌─────────────────┐
│  优化策略        │
└─────────────────┘
        ↓
┌─────────────────┐
│  总结经验        │
└─────────────────┘
        ↓
    返回第一步
```

---

### 3. 自主检查和修复 🔧

**原则**: 不接受人工修正，需要自主完成检查和修复循环

**原因**:
- 系统应该能够自我诊断和修复
- 人工修正是最后手段，不是常规操作
- 需要建立自动化检查机制

**应用**:
- ✅ 浏览器打开网页 → 检查原始内容
- ✅ 对比原始内容 vs 提取结果 → 定位问题
- ✅ 识别问题环节（Jina 提取? Prompt 设计?）
- ✅ 优化策略 → 重新测试
- ✅ 持续循环直到问题解决
- ❌ 不要依赖人工修正数据

**检查循环**:
```
┌─────────────────┐
│  浏览器打开网页  │
└─────────────────┘
        ↓
┌─────────────────┐
│  检查原始内容    │
└─────────────────┘
        ↓
┌─────────────────┐
│  对比提取结果    │
└─────────────────┘
        ↓
┌─────────────────┐
│  定位问题环节    │
│  - Jina 提取？  │
│  - Prompt 设计？│
│  - 解析器？     │
└─────────────────┘
        ↓
┌─────────────────┐
│  优化策略        │
└─────────────────┘
        ↓
┌─────────────────┐
│  重新测试        │
└─────────────────┘
        ↓
    返回第一步
```

## 架构原则

### 1. 原始数据优先 (Raw Data First)

**原则**: 
- ✅ **保留所有原始数据**（图片、链接、完整HTML结构）
- ✅ **提取和处理逻辑放在后续环节**（智谱批处理、解析器）
- ✅ **实现最大的灵活性**

**原因**:
- 原始数据一旦丢失，无法恢复
- 不同的处理需求可能需要不同的数据
- 后续处理可以灵活调整，但原始数据必须完整

**应用场景**:
- **Jina Reader API**: 保留所有图片、链接、完整Markdown结构
- **数据库存储**: raw_markdown 表存储完整的原始内容
- **智谱批处理**: 从原始数据中提取结构化信息

---

### 2. 数据管道架构

```
┌─────────────────┐
│  Jina Reader    │ → 保留所有原始数据（图片、链接、完整HTML结构）
│  (Raw Data)     │
└─────────────────┘
        ↓
┌─────────────────┐
│  raw_markdown   │ → 存储原始 Markdown（包含所有信息）
│  (Storage)      │
└─────────────────┘
        ↓
┌─────────────────┐
│  智谱批处理      │ → 提取结构化信息（标题、作者、地址等）
│  (Extraction)   │
└─────────────────┘
        ↓
┌─────────────────┐
│  paper_leads    │ → 存储结构化数据（提取后的信息）
│  (Storage)      │
└─────────────────┘
        ↓
┌─────────────────┐
│  评分/导出/应用  │ → 业务逻辑（评分、导出CSV、Feishu通知）
│  (Business)     │
└─────────────────┘
```

---

### 3. Jina Reader API 参数配置

**原则**: 保留所有原始数据

```python
headers = {
    'X-Retain-Links': 'all',    # ✅ 保留所有链接（图片链接、邮箱链接等）
    'X-Retain-Images': 'all',   # ✅ 保留所有图片
    'X-Remove-Selector': (
        'nav, aside, footer, .sidebar, '
        '.advertisement, .comments, '
        '.related-articles, .social-share'
    ),  # ✅ 只删除导航栏、广告等无关元素
}
```

**注意**:
- ❌ **不要**在 Jina Reader 环节删除任何数据
- ✅ **可以**删除导航栏、广告等无关元素（这些不影响数据提取）

---

### 4. 智谱批处理 Prompt

**原则**: 从原始数据中提取结构化信息

**Prompt 设计要点**:
- 从 Markdown 文本中提取作者姓名（忽略图片链接）
- 从 Markdown 文本中提取邮箱（忽略 `mailto:` 链接）
- 从 Markdown 文本中提取地址（忽略图片）
- 识别通讯作者标注（*、#、†等符号）
- 翻译机构地址为中文
